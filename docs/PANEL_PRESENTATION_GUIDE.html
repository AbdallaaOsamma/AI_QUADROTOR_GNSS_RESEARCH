QUICK REFERENCE — 30-Second Pitch

"My project teaches a drone to navigate autonomously in environments where GPS is unavailable — such as indoors, underground, or in areas where GPS signals are jammed. Instead of programming it with rigid rules, I used Reinforcement Learning: an AI technique where the drone learns by trial and error in a realistic flight simulator. After millions of practice flights in simulation, the drone develops a control policy — a decision-making brain — that can navigate around obstacles using only a depth camera and velocity sensors, with no GPS at all. The goal is to produce a policy that can eventually be transferred to a real drone."

SECTION 1 — THE PROBLEM: WHY DOES THIS PROJECT EXIST?

What is GNSS-Denied?

GNSS stands for Global Navigation Satellite System — the family of technologies that includes GPS (American), GLONASS (Russian), Galileo (European), and BeiDou (Chinese). All modern autonomous drones rely on GNSS to know where they are, where they're going, and how to correct their flight path.

The problem: GNSS signals are extremely weak radio waves that travel from satellites 20,000 km away. They are easy to lose and easy to disrupt:

  • Indoors: Satellite signals cannot penetrate concrete buildings. Any indoor drone is immediately blind.
  • Underground: Tunnels, mines, subway systems — zero GPS.
  • Urban canyons: Tall buildings reflect and scatter GPS signals (multipath error), making position estimates unreliable in dense cities.
  • Electronic jamming: A cheap GPS jammer (available online) can deny GPS to all drones within hundreds of metres. This is a serious military and security concern.
  • Contested environments: Military operations, search and rescue in disaster zones, or inspection of metal structures — all require navigation without GPS.

Why Existing Solutions Fail

Today's commercial drones — DJI, Parrot, and others — simply fall back to "position hold" mode when GPS is lost. If signal is not recovered quickly, they land (or crash). There is no autonomous navigation capability without GPS.

Some drones carry pre-built maps and try to match their camera feed against those maps (Visual-Inertial Odometry, VIO). This works in known environments but fails the moment the drone encounters a space it has never seen before. It also requires extensive pre-mapping, which is impossible in disaster response or military scenarios.

The Core Question This Project Answers

Can a drone learn — through experience in a simulator — to navigate through unknown spaces using only a depth camera and inertial sensing, with no GPS whatsoever?

Our answer is yes. And we use Reinforcement Learning to achieve it.

SECTION 2 — OUR SOLUTION AT A HIGH LEVEL

We built a complete software system that trains a drone's "brain" entirely inside a physics simulator called Microsoft AirSim. The brain is a deep neural network — a mathematical function with millions of adjustable parameters — that takes as input what the drone can see (a depth image from a forward-facing camera) and what the drone can feel (its velocity in body-frame coordinates from simulated inertial sensors). The network outputs velocity commands: how fast to fly forward, sideways, and how fast to rotate. The network is trained using Proximal Policy Optimisation (PPO), a Reinforcement Learning algorithm that improves the network by rewarding good behaviour (flying forward without crashing) and penalising bad behaviour (collisions, jerky motion). After millions of training steps across randomised environments, the network converges on a control policy that navigates reliably without GPS. The trained policy is then exported in the ONNX format for deployment on physical drone hardware.

SECTION 3 — THE SIMULATION ENVIRONMENT: MICROSOFT AIRSIM

What is AirSim?

AirSim (Aerial Informatics and Robotics Simulation) is an open-source drone and car simulator built by Microsoft Research on top of Unreal Engine 4. It provides:

  • Photorealistic rendering (important for visual realism)
  • Accurate physics simulation (aerodynamics, motor thrust, drag, collisions)
  • A Python API for sending flight commands and receiving sensor data

Think of it as a video game engine repurposed for robotics research. Instead of a player controlling the drone with a joystick, our Python code sends velocity commands and reads back camera images and sensor data.

Why Not Train on a Real Drone?

  1. Safety: A learning drone crashes constantly in early training. You cannot do this with a real drone without destroying it within minutes.
  2. Speed: Training requires ~1 million flight steps. At 10 Hz, that is 100,000 seconds of flight time. Simulation can be parallelised and runs faster than real time.
  3. Reproducibility: In simulation, we can reset to identical starting conditions and run controlled experiments.

Lockstep Mode — Why It Matters

We use lockstep mode: the simulation is paused after every control step. The sequence is:

PAUSE simulation
Send velocity command
RESUME simulation for exactly dt = 0.1 seconds
PAUSE simulation again
Read sensors (depth image, velocity)

From src/environments/airsim_env.py, lines 405–417:

self.client.moveByVelocityZBodyFrameAsync(...)   # Send command to drone
self.client.simContinueForTime(self.dt)           # Run physics for exactly 0.1s
self.client.simPause(True)                        # Pause again — wait for Python

The result is perfectly deterministic training — the same random seed always produces the same trajectory. This is essential for reproducible science.

Episode Reset — Starting Each Training Flight

Each episode reset sequence (lines 351–393):

  1. Unpause the simulation (blocking calls fail on a paused sim)
  2. Call client.reset() to wipe the drone's state
  3. Use simSetVehiclePose(ignore_collision=True) to teleport the drone to spawn position
  4. Enable API control, arm the motors, take off
  5. Fly to target altitude (3 m)
  6. Record current collision timestamp — to filter stale collision flags later
  7. Re-enter lockstep mode: simPause(True)
  8. Apply domain randomization

SECTION 4 — THE OBSERVATION PIPELINE: WHAT THE DRONE PERCEIVES

The Sensor Philosophy: No GPS, No Maps

A human pilot navigating without GPS uses two senses: vision (what is in front of me?) and proprioception (what is my body doing?). Our drone does exactly the same. Its observation — the information fed to the neural network at every time step — consists of exactly two things:

  1. A depth camera image (vision)
  2. A velocity vector in body frame (proprioception)

Nothing else. No GPS coordinates. No pre-built map. No ground truth position.

Image Processing Pipeline (lines 200–221)

responses = self.client.simGetImages([
    airsim.ImageRequest("0", airsim.ImageType.DepthPerspective, True, False)
])
img1d = np.array(responses[0].image_data_float, dtype=np.float32)
img_depth = img1d.reshape(height, width)
img_depth = cv2.resize(img_depth, (84, 84))        # Resize to 84x84
img_depth = np.clip(img_depth, 0, 20.0) / 20.0    # Clip at 20m, normalise to [0,1]
# Add sensor noise if domain randomization enabled:
noise = np.random.normal(0, 0.02, img_depth.shape)
img_depth = np.clip(img_depth + noise, 0.0, 1.0)

Why 84×84? This is the established size from DeepMind's Atari DQN work. Small enough for fast GPU processing, large enough to capture spatial structure (corridors, obstacles). Higher resolution gives diminishing returns at much higher compute cost.

Frame Stacking — Giving the Drone Memory

A single depth frame is a photograph — it shows WHERE things are but not HOW they move. We stack the last 4 frames together:

  • Shape goes from (84, 84, 1) to (84, 84, 4)
  • The network can compare consecutive frames to infer motion (like optical flow)
  • Essential for dynamic obstacle avoidance

Analogy: Catching a thrown ball. From a single photograph, you cannot determine its trajectory. Watch four frames in sequence, and the trajectory becomes obvious.

# In train.py, line 252:
train_env = VecFrameStack(train_env, n_stack=4, channels_order="last")

Body-Frame Velocity (lines 223–250)

Why body frame? The drone cannot know its world-frame velocity without GPS — it doesn't know which direction is "north." But it CAN measure how fast it is moving relative to itself (body frame) using an IMU. This is the key insight: we only use sensors that work without GPS.

v_global = [vx_world, vy_world, vz_world]   # From AirSim physics
yaw = euler_angles(orientation)[2]            # Drone's current heading
R_yaw = [[cos(-yaw), -sin(-yaw), 0],
          [sin(-yaw),  cos(-yaw), 0],
          [0,          0,         1]]
v_body = R_yaw @ v_global                     # Rotate to body frame
# Output: [vx_forward, vy_lateral, yaw_rate]

SECTION 5 — THE REWARD FUNCTION: TEACHING THE DRONE WHAT "GOOD" MEANS

The Reward Signal

RL works like training a dog: good behaviour gets a reward, bad behaviour gets a penalty. Over millions of repetitions, the agent learns to take actions that maximise its accumulated reward. We cannot hand-write a rule for every situation — instead we define a reward function that SCORES behaviour at every step, and let the algorithm figure out the rules.

Component 1 — Progress Reward

reward_progress = 0.5 × vx_body

If the drone flies forward at 3 m/s → reward = +1.5 per step. Hovering → reward = 0. Flying backward → small negative. Why: directly incentivises the primary goal — moving through the environment.

Component 2 — Collision Penalty

reward_collision = -100    (episode ends immediately)
# Collision detection with stale-flag filtering (lines 429-433):
has_collided = (col_info.has_collided and col_info.time_stamp != self._last_col_ts)

Why -100? The agent perceives crashing as catastrophically worse than slow/cautious flight. Without this, the agent would learn to fly at maximum speed into walls. The stale-flag filtering prevents false collisions from the previous episode contaminating the current one.

Component 3 — Smoothness Penalty

reward_smoothness = -0.1 × ||action_t - action_{t-1}||

Penalises sudden changes in velocity commands. ME analogy: Smooth driving uses less fuel and causes less mechanical wear than repeatedly flooring the accelerator and braking hard. The smoothness penalty teaches the drone the equivalent of smooth throttle control — which also produces better sim-to-real transfer because real actuators cannot execute instantaneous velocity jumps.

Component 4 — VIO Drift Penalty (DR config)

reward_drift = -0.1 × drift_error    # drift_error = |VIO_position - true_position|

When domain randomization is enabled, an additional penalty proportional to accumulated VIO position error teaches the agent to take actions that minimise estimation error.

The Full Reward Formula

reward = (0.5 × vx_body)
       + (-100 if collision else 0)
       + (-0.1 × ||action_t - action_{t-1}||)
       + (-0.1 × drift_error)   # only with DR enabled

SECTION 6 — WHY REINFORCEMENT LEARNING?

Why Not PID?

As a Mechanical Engineer, you know PID well. PID is an excellent feedback controller for well-defined setpoint tracking problems. It fails here for three reasons:

  1. No setpoint: Without GPS, we cannot define a position setpoint. PID cannot navigate without knowing "here" and "target."
  2. No perception: PID has no awareness of obstacles. Obstacle avoidance requires a separate, hand-engineered perception stack on top of PID — an ever-growing pile of rules.
  3. No adaptation: PID gains are tuned for a specific drone model. Change the payload, air density, or motor wear, and the controller degrades.

The PID controller in this project (src/control/pid.py) exists only as a baseline comparison for evaluation — to show what rule-based control achieves vs what RL achieves.

Why Not Supervised Learning?

  1. No training data: You would need expert pilots to fly through hundreds of different environments — every obstacle configuration, every corridor width. This does not exist.
  2. Distribution shift: Even with expert data, the model only works on situations that look like the training data. Any novel scenario causes failure.
  3. Cannot exceed the expert: Imitation learning can only be as good as the demonstrations. RL can discover strategies better than any human expert.

Why Not Model-Based Planning?

Model-based control builds an explicit world model and plans paths through it (A*, RRT). This requires a pre-built map. In GNSS-denied unknown environments, there is no map. Real-time SLAM is computationally expensive and fails in featureless environments.

Why RL Wins

RL requires no labelled data, no pre-built map, and no accurate dynamics model. It learns from experience — trial, error, and reward.

The RL Loop:

1. Drone observes current state (depth image + velocity)
2. Neural network (the "policy") outputs an action (velocity command)
3. Action is executed in simulation
4. Environment computes reward (+good, -bad)
5. Reward signal adjusts the neural network's weights
6. Repeat from step 1, one million times

Why RL generalises: Because we use domain randomization, the agent trains across thousands of different starting positions, obstacle configurations, and sensor noise levels. It learns a STRATEGY, not a specific path.

SECTION 7 — WHY PPO SPECIFICALLY?

What is a "Policy"?

A policy is a mathematical function: policy(observation) → action. It maps what the drone perceives to what the drone should do. The policy is a neural network — the RL algorithm's job is to find the network weights that make this function as good as possible.

What is Gradient Descent?

Neural network training uses gradient descent. The "loss" is how bad the policy currently is. The gradient is the direction of steepest improvement. We take small steps in that direction.

ME analogy: You are blindfolded on a hilly landscape, trying to reach the lowest valley. You feel the slope underfoot (the gradient) and take a step downhill. Repeat, and you converge to the valley. If you take too large a step, you overshoot the valley and end up on the other side of a hill — WORSE than before. This is the catastrophic update problem that PPO solves.

PPO's Key Innovation — The Clipped Objective

ratio = new_policy_probability / old_policy_probability
objective = min(ratio × advantage, clip(ratio, 0.8, 1.2) × advantage)

At each update, PPO is not allowed to change the policy by more than 20% (clip_range = 0.2). This is a hard constraint that prevents catastrophic updates.

ME analogy: Adjusting a control valve. You calculated that opening it by 80% maximises flow — but doing it that fast would surge and break the system. So you impose a rule: maximum 20% change per adjustment. Next adjustment, another 20% closer to optimal. PPO is this rule, applied to neural network weight updates.

SECTION 8 — THE NEURAL NETWORK ARCHITECTURE

What is a Neural Network, From Scratch?

A neural network is a mathematical function with adjustable parameters (weights). Each layer takes inputs, multiplies them by weights, sums them, applies a non-linear function (ReLU: max(0, x)), and passes to the next layer.

ME analogy: Think of it as a very complex transfer function. Input goes in, emerges transformed through many stages, producing an output. The difference: the parameters ADAPT through training to produce the desired input-output relationship.

Why non-linear functions? A sequence of purely linear operations collapses to a single linear transformation, no matter how many layers. Non-linearities give neural networks the power to represent complex mappings like "navigate around obstacles."

CNN — Convolutional Neural Network

A CNN uses sliding filter windows that scan across an image looking for patterns. Each filter becomes a "feature detector" — one might detect edges, another open corridors. The CNN learns WHICH patterns are important during training.

NatureCNN — The Architecture We Use

NatureCNN is from DeepMind's 2015 paper "Human-level control through deep reinforcement learning" — the same architecture that learned to play 49 Atari video games from pixels alone.

Input: (84, 84, 4)  ← 4 stacked depth frames
  ↓
Conv2D(32 filters, 8×8 kernel, stride 4)   ← large-scale features
  ↓ ReLU
Conv2D(64 filters, 4×4 kernel, stride 2)   ← medium-scale features
  ↓ ReLU
Conv2D(64 filters, 3×3 kernel, stride 1)   ← fine-grained features
  ↓ ReLU
Flatten → Linear(512)
  ↓
Output: 512-dimensional image feature vector

MLP for Velocity

The velocity observation [vx, vy, yaw_rate] is just 3 numbers. It has no spatial structure — no need for convolutions. A simple fully connected (MLP) network processes it:

Input: (3,)  ← [vx, vy, yaw_rate]
  ↓ Linear(64) → ReLU
Output: 64-dimensional velocity feature vector

Late Fusion — Why Combine at the End?

Early fusion: mix raw pixels and velocity numbers before processing. Problem: the CNN expects to slide spatial filters over image data. Mixing in velocity numbers at the pixel level breaks this spatial interpretation.

Late fusion: process each modality completely separately, THEN concatenate at a semantic level. The two feature vectors (spatial features + motion features) are concatenated and fed to the actor and critic heads.

Image (84,84,4) → NatureCNN → 512-dim vector ─┐
                                                 ├─ Concatenate (576-dim)
Velocity (3,) → MLP → 64-dim vector  ──────────┘
                                                 ↓
                                      Actor head → action (velocity command)
                                      Critic head → value estimate (training only)

MultiInputPolicy — The stable-baselines3 Name

MultiInputPolicy is stable-baselines3's name for a policy that accepts a Dict observation space. It automatically routes image observations through NatureCNN and non-image observations through MLP, then performs late fusion. Specified in train.py line 161:

cls("MultiInputPolicy", env=env, tensorboard_log=log_dir, ...)

SECTION 9 — DOMAIN RANDOMIZATION: BRIDGING SIM AND REALITY

The Sim-to-Real Gap

When you train in simulation and deploy on real hardware, the policy often fails. This is the sim-to-real gap — the difference between simulation's idealised world and reality's imperfect one:

  • Simulated depth: perfect. Real depth: noisy, dead pixels, reflections.
  • Simulated physics: exact. Real physics: motor vibration, wear, non-linearity.
  • Simulated VIO: perfect velocity. Real VIO: drift and bias accumulation.

If the policy learns to exploit simulation-specific artefacts (perfect readings, fixed spawn point), it fails the moment those artefacts disappear in reality.

Domain Randomization — Training on Uncertainty

Military analogy: Train soldiers in mud and rain — not just controlled indoor ranges. Build soldiers who perform under any conditions. Domain randomization is this principle applied to drone training.

What We Randomise (lines 161–195, configs/train_ppo_dr.yaml)

domain_randomization:
  enabled: true
  depth_noise_std: 0.02        # Gaussian noise on normalised depth image
  spawn_radius_m: 5.0          # Random start position within 5m radius
  vio_enabled: true
  vio_drift_std_per_step: 0.01 # Accumulated VIO drift per step (m/s)
  vio_bias_std: 0.005          # Constant per-episode IMU bias (m/s)
  flow_noise_std: 0.02         # Per-step optical flow perturbation

1. Depth Sensor Noise

noise = np.random.normal(0, 0.02, img_depth.shape)
img_depth = np.clip(img_depth + noise, 0.0, 1.0)

Simulates real Intel RealSense sensor noise (±2% of full range per pixel).

2. Spawn Position Jitter

dx = uniform(-5, 5);  dy = uniform(-5, 5)
random_yaw = uniform(-π, π)
client.simSetVehiclePose(Pose(Vector3r(dx, dy, -3.0), quaternion(yaw)), ignore_collision=True)

The agent never starts at the same place — it must learn a general navigation strategy, not memorise a path.

3. VIO Drift and Bias

See SimulatedVIO class above. Each episode gets a new random bias, different drift rate — simulating different IMU calibration states on different drones.

What is Not Yet Done (Planned Phase 2+)

  • Physics DR: varying mass, inertia, motor gain per episode
  • Wind disturbances via AirSim weather API
  • Visual degradation: simulated fog, motion blur

SECTION 11 — THE SAFETY MONITOR

Why Safety is a Separate Module

The RL policy is a learned function that can fail on inputs outside its training distribution. The safety monitor is a deterministic, rule-based filter that sits between the RL policy and the flight controller, operating only during deployment.

Architecture: RL Policy → velocity command → Safety Monitor → safe command → Flight Controller

ME analogy: ABS + stability control in a modern car. The driver (RL) decides direction and throttle; the safety system intervenes to prevent wheel lock or dangerous speeds. The driver's intent is preserved, but dangerous commands are overridden.

The Four Safety Layers (safety_node.py, lines 42–67)

def apply_safety(vx, vy, yaw_rate, depth_m, estop, ...):
    # Layer 0: Emergency stop — HIGHEST PRIORITY
    if estop:
        return 0.0, 0.0, 0.0

    # Layer 1: Velocity clamping
    vx       = clip(vx,       -3.0, 3.0)   # Never exceed 3 m/s forward
    vy       = clip(vy,       -1.0, 1.0)   # Never exceed 1 m/s lateral
    yaw_rate = clip(yaw_rate, -45°/s, 45°/s)

    # Layer 2: Proximity braking
    if depth_m is not None and vx > 0:
        min_depth = centre_roi_min_depth(depth_m)  # Min depth in centre 30%
        if min_depth < 1.5:                         # Obstacle closer than 1.5m
            vx *= 0.2                               # Reduce forward speed to 20%

    return vx, vy, yaw_rate

SECTION 12 — SIM-TO-REAL TRANSFER

ONNX Export — The Universal Model Format

ONNX (Open Neural Network Exchange) is a standardised format for neural network models — like a PDF for AI models. It captures architecture and weights in a hardware-agnostic format supported by every major platform.

scripts/export_onnx.py: Load the trained stable-baselines3 .zip → extract PyTorch network → trace computation graph → export to .onnx. On deployment hardware, onnxruntime runs inference at full speed without requiring PyTorch or stable-baselines3.

SECTION 15 — TOUGH PANEL Q&A: 12 PREPARED ANSWERS

Use these as starting points. Rephrase in your own words. Confidence matters as much as content.

Q1: "Why not just fix the GPS or use better GPS?"
GPS is an externally-dependent, infrastructure-based system. Even the best GPS is unavailable indoors, underground, and in electronically contested environments. Modern GPS jammers cost less than £50 and can deny GPS to all drones within hundreds of metres. Military adversaries routinely jam GPS in conflict zones. Better GPS hardware does not solve these fundamental limitations — the signal physically cannot penetrate concrete, and jamming is a trivial attack. The only robust solution for autonomous operation in these environments is onboard sensing that requires no external infrastructure. Our system uses exactly that.
Q2: "How is this better than a PID controller?"
A PID controller is excellent for well-defined setpoint tracking. But it has three fundamental limitations here. First, without GPS we cannot define a position setpoint — PID cannot navigate when it doesn't know where it is. Second, PID has no obstacle perception — it would fly straight into walls without a separate, hand-engineered avoidance system on top. Third, PID is tuned for one specific drone model and does not adapt to changes in payload, air density, or motor wear. Our RL policy learns from experience across thousands of scenarios, incorporates visual obstacle perception directly, and produces an adaptive control strategy that generalises to unseen environments. PID is a hammer; this problem is a screw.
Q3: "What is the sim-to-real gap, and how do you address it?"
The sim-to-real gap is the difference between idealised simulation and messy reality. In simulation: depth images are perfect, physics is exact, sensors have no noise. In reality: depth cameras have noise and dead pixels, motors vibrate, IMUs drift. A policy trained on perfect simulation often fails on real hardware because it learned to rely on simulation-specific artefacts. We address this with domain randomization — deliberately adding noise and variation during training so the policy learns behaviours robust to imperfection. Additionally, we chose depth (not RGB) because simulated depth images look much more like real depth images. Finally, ONNX export ensures the trained model runs identically on any target hardware.
Q4: "Why depth camera and not RGB?"
Three reasons. First, depth is lighting-invariant — it uses infrared and works in darkness, bright sunlight, or flickering light, where RGB fails. Second, depth is texture-invariant — consistent readings for walls regardless of colour or pattern, so the policy doesn't need to learn "red walls and white walls are both obstacles." Third, and most critically for this project, the sim-to-real gap for depth is far smaller than for RGB. Simulated depth images look qualitatively similar to real RealSense images. Simulated RGB looks nothing like real images — wrong colours, textures, lighting. Choosing depth was a deliberate architectural decision to maximise sim-to-real transferability.
Q5: "What does frame stacking give you that a single frame doesn't?"
A single depth frame is a snapshot — it shows where obstacles are, not how they or the drone are moving. Consider catching a thrown ball: from a single photograph, you cannot determine its trajectory. Watch four frames, and the trajectory is obvious. Our drone faces the same problem: to predict whether it is on a collision course, it needs the rate of change of obstacle positions relative to itself. Four stacked frames provide this temporal information. The neural network learns to compare consecutive frames and detect motion — effectively computing optical flow internally without an explicit calculation. Remove frame stacking (one of our ablation experiments), and we expect the collision rate to increase measurably.
Q6: "Why PPO over SAC? SAC is often said to be more sample-efficient."
SAC is more sample-efficient — it reuses experience from a replay buffer. But sample efficiency is not our primary constraint — we generate experience cheaply in simulation. Our primary constraint is training stability. PPO's clipped objective guarantees no catastrophic policy updates — the policy cannot change by more than 20% in any update. SAC is more sensitive to hyperparameter choices, particularly the entropy coefficient. For Phase 1 infrastructure building and validation, PPO's stability was the correct engineering trade-off. SAC is deliberately planned for Phase 2: once PPO results are established, we run the same evaluation protocol on SAC and apply ANOVA and paired t-tests to determine if any performance difference is statistically significant.
Q7: "What would happen without domain randomization?"
Without DR, the policy trains on perfect sensor readings, a fixed spawn position, and zero VIO drift. It would likely learn to exploit simulation-specific artefacts — always expecting a perfect depth image, always starting at exactly the same position. In evaluation in a different environment or on real hardware, those assumptions are violated. The most likely failure mode: the policy navigates perfectly in the training scenario but crashes immediately in any other room layout. This is overfitting to the training distribution. Domain randomization prevents this by forcing the policy to learn robust, generalised strategies rather than environment-specific shortcuts. That's why we have dedicated ablation experiments to quantify exactly how much DR contributes to generalisation.
Q8: "How do you know your reward function is correct?"
Reward function design is the hardest part of applied RL, and I'll be honest that "correct" is a high bar. We validate it three ways. First, theoretical reasoning: each component has a clear causal link to desired behaviour — progress reward encourages movement, collision penalty enforces safety, smoothness penalty promotes deployability. Second, ablation studies: we remove each component and measure performance. If removing the smoothness penalty produces measurably higher action jerk, the penalty was doing its job. Third, reward sweep experiments: we train with three weight profiles (default, aggressive, cautious) to understand sensitivity to specific values. We don't claim it is optimal — reward engineering is an active research area. We claim it is motivated, justified, and empirically tested.
Q9: "What are the limitations of your approach?"
I'll be direct about four key limitations. First, we have not yet run the full training campaigns — results in this presentation are from infrastructure validation. Second, our VIO pipeline is simulated, not real — we inject synthetic drift into ground-truth velocity, which is a simplification of a real VIO system. Third, our domain randomization covers sensor noise and spawn jitter but not physics parameters — we don't vary mass, inertia, or motor characteristics per episode. Fourth, the hardware prototype has not been built — ONNX export exists but the complete MAVLink/ROS2 integration and physical flight testing are Phase 3 work. These are documented known limitations that define our post-FYP roadmap.
Q10: "What's next after the FYP?"
Five clear next steps: Run the full training campaigns — 1M steps for PPO, then SAC, collect evaluation metrics for comparison. Implement SAC and run the PPO-vs-SAC comparison with statistical analysis (ANOVA, paired t-tests). Complete physics domain randomization — varying mass, inertia, and motor parameters per episode. Replace the simulated VIO with a real VIO pipeline in the simulator loop, enabling proper localisation drift evaluation. Build the physical hardware prototype and attempt sim-to-real transfer using the ONNX-exported policy on the Jetson Xavier NX.
Q11: "What is PPO's clipped objective in simple terms?"
Think of adjusting a control valve. You calculated that opening it by 80% maximises flow — but doing it that fast would surge and break the system. So you impose a rule: maximum 20% change per adjustment cycle. Next adjustment, another 20% closer to optimal. PPO is this rule applied to neural network weight updates. At each training step, PPO calculates how much to change the policy. If the change exceeds 20%, it is clipped to exactly 20%. This prevents catastrophically large policy updates — which could suddenly make the drone fly dangerously — and ensures training remains stable over one million steps.
Q12: "Why 84×84 pixels? Why not higher resolution?"
84×84 is the resolution established by DeepMind's Atari DQN paper in 2015 and validated across hundreds of visual RL tasks since. For navigation, the relevant information in a depth image is: where are the open spaces, where are the walls, how wide is the corridor? This spatial information is fully captured at 84×84. Going to 256×256 increases the input size by nine times, the convolutional processing cost by nine times, and slows training significantly — with negligible benefit for a navigation task. We also configured AirSim's camera to output 84×84 directly in settings_training.json, eliminating the resizing overhead entirely. It is a deliberate engineering decision: the minimum resolution that preserves navigation-relevant features at maximum training speed.
